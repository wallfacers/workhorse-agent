# Feature Specification: Agent 编排能力升级（后台委派、溢出自愈、定时调度、活动上报）

**Feature Branch**: `001-agent-orchestration`

**Created**: 2026-07-20

**Status**: Draft

**Input**: User description: "借鉴 grok-cli 的 agent 设计，为 workhorse-agent 增加四项能力：后台只读委派子代理（delegate/delegation_read/delegation_list 工具、结果落盘、完成通知注入会话历史、禁止嵌套）、上下文溢出自愈（context-limit 错误触发放宽参数的强制 compaction 并整轮重试一次）、进程内 cron scheduler（schedule_create/list/remove/read_log 工具、分钟级 tick、同分钟去重、运行日志）、子代理活动实时上报（工具调用翻译为人类可读状态经 SSE subagent_status 事件上报）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 后台只读委派子代理 (Priority: P1)

用户让主 agent 处理一个需要大量代码调研的任务。主 agent 通过 `delegate` 工具把只读的探索型调研派给一个后台子代理，自己继续推进当前工作，不被阻塞。后台调研完成后，结果被持久化保存；用户下一次发消息时，主 agent 会先"看到"一条完成通知（含标题与摘要），并可通过 `delegation_read` 工具取回完整结果，将其纳入后续工作。

**Why this priority**: 这是四项能力中对主 agent 工作模式改变最大的一项——把"串行等待调研"变为"并行推进"，直接提升长任务的端到端效率；且只读约束保证了后台任务永远不会与前台产生文件写冲突，风险可控。

**Independent Test**: 在一个会话中让主 agent 委派一个后台调研任务，验证：委派立即返回且主 agent 可继续响应；后台任务完成后结果落盘可读；下一条用户消息处理前通知恰好注入一次；`delegation_read`/`delegation_list` 可检索。

**Acceptance Scenarios**:

1. **Given** 一个正在进行的会话，**When** 主 agent 调用 `delegate` 发起后台调研，**Then** 调用立即返回一个可读性好的委派 ID 与"完成后将收到通知"的说明，主 agent 可继续执行后续工具调用。
2. **Given** 一个已完成的后台委派，**When** 用户发送下一条消息，**Then** 该轮处理开始前，完成通知（含委派 ID、标题、摘要、取回方式）作为会话上下文注入恰好一次，后续消息不再重复注入。
3. **Given** 一个已完成的后台委派，**When** 主 agent 调用 `delegation_read` 并传入该 ID，**Then** 返回完整的调研结果全文。
4. **Given** 一个仍在运行的后台委派，**When** 主 agent 调用 `delegation_read`，**Then** 返回"仍在运行，请继续其他工作"的提示而非阻塞等待。
5. **Given** 一个后台委派子代理，**When** 它尝试调用任何具有写入或执行副作用的工具（写文件、执行 shell、再次委派），**Then** 该工具不在其可用工具面内，调用不可能发生。
6. **Given** 一个后台委派子代理正在运行，**When** 它尝试再发起嵌套的后台委派，**Then** 请求被拒绝并返回明确的错误说明。
7. **Given** 一个后台委派执行失败（模型错误、超时等），**When** 用户发送下一条消息，**Then** 失败通知（含错误摘要）同样注入一次，且 `delegation_read` 返回失败详情。

### User Story 2 - 上下文溢出自愈 (Priority: P2)

用户在一个长会话中继续提问。请求因累计上下文超过模型窗口而被 provider 拒绝。系统不再把错误直接抛给用户，而是自动以更激进的参数压缩历史、重试本轮请求；用户只感知到一次正常的（稍慢的）回复，会话不中断。

**Why this priority**: 消灭一类现存的硬失败（长会话必然撞上下文墙），实现成本低、收益确定；排在委派之后是因为它是"止损"而非"新能力"。

**Independent Test**: 构造一个接近上下文上限的会话，触发 provider 的 context-limit 错误，验证系统自动压缩并重试成功；再构造压缩后仍超限的极端场景，验证只重试一次后如实报错。

**Acceptance Scenarios**:

1. **Given** 一个上下文接近模型窗口上限的会话，**When** provider 返回 context-limit 类错误且本轮尚未产生任何输出，**Then** 系统以放宽参数（保留的近期上下文减半，且不低于一个安全下限）强制执行一次历史压缩，并自动重试整轮请求。
2. **Given** 已经历过一次溢出自愈重试的当前轮次，**When** 重试后再次收到 context-limit 错误，**Then** 系统不再重试，向用户返回明确的错误信息。
3. **Given** 本轮已经向用户流式输出了部分内容后才发生错误，**When** 判定是否自愈重试，**Then** 不进行自动重试（避免重复输出），按既有错误路径处理。
4. **Given** 一次成功的自愈重试，**When** 查看会话事件流，**Then** 压缩发生这一事实可被观察到（有对应事件/日志），用户收到的最终回复完整且无重复。

### User Story 3 - 进程内定时调度 (Priority: P3)

用户对 agent 说"每个工作日早上 9 点帮我跑一次依赖安全检查"。agent 通过 `schedule_create` 工具创建一个 cron 定时任务。服务器常驻进程按分钟检查到期任务，到点自动以无人值守方式运行一个新会话执行既定指令，并保留运行日志。用户随时可通过 agent 查看、删除计划或翻阅历史运行日志。

**Why this priority**: 赋予 agent "自助排班"的全新能力维度（时间循环），价值高但属于增量能力，不影响既有交互路径。

**Independent Test**: 创建一个一分钟后到期的一次性计划与一个 cron 计划，验证到点触发、同分钟不重复触发、日志可读、删除后不再触发。

**Acceptance Scenarios**:

1. **Given** 用户要求定期执行某项任务，**When** 主 agent 调用 `schedule_create`（支持 cron 表达式与一次性两种），**Then** 计划被持久化保存，重启服务后依然存在，并返回计划 ID 与下次预计触发说明。
2. **Given** 一个已启用的 cron 计划，**When** 服务器时钟进入匹配的分钟，**Then** 该计划在这一分钟内恰好触发一次（同分钟去重），以无人值守模式启动一次新会话执行既定指令。
3. **Given** 一次定时触发的运行，**When** 运行结束，**Then** 本次运行的输出被写入该计划专属的运行日志，`schedule_read_log` 可读取最近若干次日志。
4. **Given** 一个一次性计划已触发过，**When** 到达后续任意时间，**Then** 它不再触发。
5. **Given** 主 agent 调用 `schedule_list`，**When** 存在若干计划，**Then** 返回每个计划的 ID、指令摘要、触发规则、启用状态与最近一次运行时间。
6. **Given** 主 agent 调用 `schedule_remove` 删除计划，**When** 到达原定触发时间，**Then** 不再触发。
7. **Given** 一次无人值守运行中出现需要人工审批的权限请求，**When** 无人响应，**Then** 按既有权限超时语义处理（超时拒绝并可观察），运行不会无限挂起。

### User Story 4 - 子代理活动实时上报 (Priority: P4)

用户在客户端界面上观察主 agent 工作。当主 agent 把任务派给子代理时，界面不再是长时间的"黑盒等待"，而是持续显示子代理当前在做什么——每次工具调用被翻译成一行人类可读的状态（如"读取 internal/agent/loop.go"、"搜索 'permission timeout'"）。

**Why this priority**: 纯体验增强，改动面小；依赖子代理机制先就位，故排最后。

**Independent Test**: 触发一次子代理任务，订阅会话 SSE 流，验证子代理每次工具调用都产生一条人类可读的 `subagent_status` 事件，任务结束后收到清空状态的事件。

**Acceptance Scenarios**:

1. **Given** 主 agent 正在运行一个子代理任务，**When** 子代理发起一次工具调用，**Then** 会话 SSE 流上出现一条 `subagent_status` 事件，载荷包含子代理类型、任务描述与一行人类可读的当前活动。
2. **Given** 子代理任务结束（成功或失败），**When** 观察 SSE 流，**Then** 收到一条状态清空事件，客户端可据此移除状态显示。
3. **Given** 活动描述来源于任意工具参数，**When** 参数超长或包含多行内容，**Then** 上报的活动行被截断为有界长度的单行文本。

---

### Edge Cases

- 后台委派运行期间服务器重启：委派记录状态可被识别为中断/失败，通知照常（作为失败）注入，不产生僵尸"running"状态永久残留。
- 多个后台委派同时完成：通知按稳定顺序逐条注入，每条恰好一次。
- 后台委派并发上限：超出上限的委派请求被拒绝并提示稍后再试，而不是无限排队。
- 溢出自愈时会话历史本身极小（无可压缩内容）：跳过压缩、不重试，直接报错。
- cron 表达式非法：`schedule_create` 拒绝并返回可理解的校验错误。
- 服务器停机跨过触发时间点：错过的触发不补跑（无追赶语义），下个匹配时间正常触发。
- 定时任务的工作目录在触发时已不存在：该次运行失败并记入日志，计划保持存在。
- 同一会话内 `delegation_read` 传入不存在的 ID：返回"未找到，可用 delegation_list 查看"的提示。

## Requirements *(mandatory)*

### Functional Requirements

**后台只读委派**

- **FR-001**: 主 agent MUST 能通过 `delegate` 工具发起后台委派，提供任务描述与详细指令；调用立即返回委派 ID 与状态说明，不阻塞当前轮次。
- **FR-002**: 后台委派子代理的工具面 MUST 仅包含只读能力（读取、搜索类），不包含任何写文件、执行命令、再委派或定时调度能力。
- **FR-003**: 系统 MUST 拒绝嵌套委派：后台子代理不能再发起后台委派。
- **FR-004**: 委派的元数据（ID、描述、状态、起止时间、错误）与结果全文 MUST 持久化，服务重启后仍可检索。
- **FR-005**: 委派完成（或失败）后，系统 MUST 在下一次用户消息处理开始前，把包含 ID、标题、摘要与取回方式的通知注入该会话上下文，且每个委派恰好通知一次。
- **FR-006**: 主 agent MUST 能通过 `delegation_read`（按 ID 取全文）与 `delegation_list`（列出状态与摘要）检索委派；读取运行中的委派返回非阻塞的提示。
- **FR-007**: 系统 MUST 对同时运行的后台委派数量设置上限，超限请求被明确拒绝。
- **FR-008**: 委派 ID MUST 为人类可读、可口头转述的短标识，且在其作用域内唯一。

**上下文溢出自愈**

- **FR-009**: 当一轮请求因 provider 的 context-limit 类错误失败、且本轮尚未向用户输出任何内容时，系统 MUST 以放宽参数强制执行一次历史压缩并自动重试整轮，每轮至多自愈一次。
- **FR-010**: 放宽参数 MUST 表现为：保留的近期上下文预算减半，且不低于一个固定安全下限。
- **FR-011**: 自愈重试仍失败、或本轮已产生部分输出、或历史无可压缩内容时，系统 MUST 按既有错误路径向用户如实报错，不再重试。
- **FR-012**: 自愈过程中发生的压缩 MUST 可被观察（事件或日志），且压缩结果按既有压缩语义持久化。

**进程内定时调度**

- **FR-013**: 主 agent MUST 能通过 `schedule_create` 创建定时计划，支持 cron 表达式（重复）与指定时刻（一次性）两种触发规则；非法规则被拒绝并返回校验错误。
- **FR-014**: 计划 MUST 持久化（含指令、触发规则、工作目录、启用状态、最近触发时间），服务重启后自动恢复调度。
- **FR-015**: 常驻服务 MUST 以分钟粒度检查到期计划；同一计划在同一分钟内 MUST 至多触发一次；错过的时间点不补跑。
- **FR-016**: 触发 MUST 以无人值守方式启动一次新会话执行计划指令，运行输出写入该计划专属日志；`schedule_read_log` 可读取最近日志。
- **FR-017**: 主 agent MUST 能通过 `schedule_list` / `schedule_remove` 查看与删除计划；删除立即生效。
- **FR-018**: 无人值守运行中的权限请求 MUST 沿用既有权限体系与超时语义，不得无限挂起，也不得绕过权限检查。
- **FR-019**: 一次性计划触发一次后 MUST 自动失效。

**子代理活动上报**

- **FR-020**: 子代理每次工具调用 MUST 被翻译为一行有界长度的人类可读活动描述。
- **FR-021**: 活动描述 MUST 以 `subagent_status` 事件（含子代理类型、任务描述、当前活动）实时发布到所属会话的事件流；任务结束时发布状态清空事件。
- **FR-022**: 活动上报 MUST 是尽力而为的：上报失败不得影响子代理任务本身的执行与结果。

**通用约束**

- **FR-023**: 以上全部新工具 MUST 经由既有工具注册与 `allowed_tools` 过滤机制暴露，遵守既有权限检查流程。
- **FR-024**: 以上全部持久化 MUST 使用既有的本地存储机制；事件 MUST 遵循既有事件流协议（追加式、可重放）。

### Key Entities

- **Delegation（委派记录）**: 一次后台调研任务。属性：人类可读 ID、所属会话、任务描述、详细指令、状态（running/complete/error）、起止时间、标题、摘要、结果全文、是否已通知。
- **Schedule（定时计划）**: 一条自动化排班。属性：ID、名称、指令、触发规则（cron 或一次性时刻）、工作目录、启用状态、创建时间、最近触发时间。
- **ScheduleRun（计划运行日志）**: 计划的一次触发记录。属性：所属计划、触发时间、运行输出日志、结束状态。
- **SubagentStatus（子代理状态）**: 瞬时状态快照。属性：子代理类型、任务描述、当前活动一行文本；生命周期与子代理任务一致，不持久化。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主 agent 发起后台委派后，能在 2 秒内继续执行下一步工作（不被委派阻塞）。
- **SC-002**: 100% 的已完成委派在下一条用户消息处理前恰好收到一次通知（不丢失、不重复）。
- **SC-003**: 在会话历史存在可压缩空间的前提下，context-limit 失败的轮次经自愈后成功完成的比例达到 95% 以上；用户在这些轮次中不会看到裸露的 provider 错误。
- **SC-004**: 定时计划在系统正常运行期间的触发准时率 100%（误差不超过一个检查周期，即 1 分钟），且同分钟重复触发次数为 0。
- **SC-005**: 子代理任务执行期间，客户端每次子代理工具调用都能收到对应的状态事件，观察者可以在任意时刻回答"子代理现在在干什么"。
- **SC-006**: 全部四项能力启用后，既有测试套件与既有 SSE 客户端（不认识新事件类型的旧客户端）行为不受破坏。

## Assumptions

- 后台委派子代理与主会话共享同一进程与同一套 provider 配置，采用进程内并发而非独立子进程（服务本身是常驻的，这与借鉴对象的独立进程方案不同，但达成同样的隔离目标：只读工具面即隔离边界）。
- 后台委派并发上限默认取一个小的固定值（建议 4），暂不做配置项；超限直接拒绝。
- 委派结果无自动清理策略（保留全部历史记录），清理留待后续需求。
- 溢出自愈的"context-limit 类错误"依靠对 provider 错误的分类识别；识别不到的错误按普通错误处理，宁可漏报不可误报。
- 定时调度运行在 serve 常驻进程内，不引入独立 daemon 进程；serve 未运行期间计划不触发也不补跑。
- 无人值守运行沿用既有的权限超时拒绝语义（permission_request_timeout），不新增"无人值守专用权限策略"；需要审批的操作在无人值守场景下自然落空为拒绝。
- cron 表达式采用标准五字段格式（分 时 日 月 周），时区取服务器本地时区。
- `subagent_status` 为新增 SSE 事件类型，旧客户端忽略未知事件类型即可，协议向后兼容。
- 现有会话/子代理（Dispatch）机制是活动上报的挂载点；本特性不改变既有子代理的调用方式。
