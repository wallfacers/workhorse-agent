## Why

`add-external-agent-tool` (prerequisite change) gives `workhorse-agent` the ability to invoke any CLI declared in a `~/.workhorse-agent/external-agents/<name>.yaml` adapter file as a first-class sub-agent. That solves the runtime question ŌĆö *how do we drive it* ŌĆö but leaves an extensibility one: *how does an adapter get written in the first place?* The MVP shipping path for new adapters is "user opens an editor and writes ~30 lines of YAML against a schema". For a project whose explicit goal is "µē®Õ▒ĢõĖŹµö╣õ╗ŻńĀüŃĆüõĖŹµö╣ķģŹńĮ«", that is a meaningful papercut.

This change closes the loop. A new built-in helper ŌĆö backed by a dedicated `adapter-generator` agent type and an `agent_setup` tool callable by the main orchestrator ŌĆö walks an installed CLI, gathers its `--help` / `--version` / man-page output, asks an LLM to produce a fully-typed adapter YAML against the schema from Change 1, runs the same smoke-test infrastructure on the candidate in a sandbox, and surfaces a diff-style approval prompt to the user. Approved adapters drop into the same `~/.workhorse-agent/external-agents/` directory the existing registry already hot-reloads. ## What Changes

- Introduce a new agent type **`adapter-generator`**, shipped as an embedded builtin, with a narrow system prompt focused on producing valid adapter YAML against the schema from `add-external-agent-tool`. Its tool surface is restricted to a small allowlist: `Bash` (read-only invocations: `which`, `<bin> --help`, `<bin> --version`, `man <bin>`, `cat <readme>`), `Read`, and a new internal-only `WriteAdapterDraft` tool that writes proposals to `<profileDir>/external-agents/.drafts/<name>.yaml` (NEVER directly to the active adapters dir).
- Add a new public tool **`agent_setup`** to the main orchestrator's surface (`internal/tools/agentsetup/`). It accepts a binary name or absolute path plus an optional `description_hint`, runs the generator subagent, then runs the same smoke-test pipeline as `add-external-agent-tool` on the draft, then emits an approval request to the user.
- Add a new `workhorse-agent setup-agent <binary>` **CLI subcommand** that performs the same flow non-interactively for explicit triggering from a terminal.
- Add **implicit-trigger detection** in the agent loop: when the model attempts to call `ExternalAgent` with an `agent_name` that does not exist in the current enum AND a binary matching that name resolves on `PATH`, the loop synthesises an `agent_setup` invocation and presents the result before failing. This is the "ńö©µłĘĶ»┤'ńö© gemini'" case from the explore session.
- Add an **approval surface** to the existing API protocol: new event `adapter_approval_request` carries the generated YAML + smoke result + diff against any prior version; new endpoint `POST /sessions/{id}/approvals/{approval_id}` accepts `approve | reject | edit` with optional edited YAML body. Approvals time out after 5 minutes (configurable) ŌåÆ reject.
- Add a new `internal/extagent/regen/` package implementing **drift detection**: at server startup, compare `<bin> --version` against the cached `provenance.tool_version` in every `llm_generated` adapter; on mismatch, emit a structured log line (no auto-regen) and surface in `GET /diagnostics`.
- Extend the adapter schema's `provenance` block usage: LLM-generated adapters MUST populate `generated_by` (model id), `generated_at` (timestamp), `tool_version` (string captured at generation time), `reviewed_by` (user id or `"user"`).
- Add **prompt templates** for adapter generation: a new `AdapterGeneration` template in `internal/prompt/builtins.go` rendering the schema + collected CLI metadata + few-shot examples.

## Capabilities

### New Capabilities
- `adapter-generation`: end-to-end LLM-driven adapter generation flow ŌĆö information collection, candidate generation, validation, sandbox smoke test, approval UX, drift detection, persistence semantics.

### Modified Capabilities
- `external-agents`: extend the approval gate from `add-external-agent-tool` ┬¦"First-invocation approval for untrusted adapters" so an adapter that was just `llm_generated` and approved through `adapter-generation` is treated as having passed first-invocation approval for the originating session (no double-prompt). All other sessions still gate normally per the existing rule.
- `api-protocol`: ADD the `adapter_approval_request` event type and the `POST /sessions/{id}/approvals/{approval_id}` endpoint; no breaking changes to existing events or endpoints.

## Impact

- **Code**: new `internal/extagent/regen/` (drift), new `internal/extagent/draft/` (atomic draft ŌåÆ approved ŌåÆ published moves ŌĆö publish + genmeta logic), new `internal/tools/extagent/drafttool/` (the `WriteAdapterDraft` tool, isolated from publish logic to avoid the awkward duplicate-`draft` package naming), new `internal/tools/agentsetup/` (`agent_setup` tool), new builtin agent type `adapter-generator` in `~/.workhorse-agent/agents/` (or embedded), new HTTP handler for approvals in `internal/api/`, three new SSE event types, new `setup-agent` and `approve` CLI subcommands in `cmd/workhorse-agent/`, one new prompt template in `internal/prompt/builtins.go`.
- **Persistence**: drafts live at `<profileDir>/external-agents/.drafts/<name>.yaml` (a hidden subdirectory the loader from Change 1 explicitly ignores). Approved adapters move via `rename(2)` to `<profileDir>/external-agents/<name>.yaml`. Pending approvals are kept in-memory in the session manager with a TTL; they do not survive server restart (a restart cancels pending approvals ŌĆö user must retry).
- **Dependencies**: none added. Reuses `gopkg.in/yaml.v3`, the JSON Schema validator from Change 1, the driver/smoke runner from Change 1, the permission package, and the existing prompt template engine.
- **Config**: optional `external_agents.generation.approval_timeout_sec` (default `300`), `external_agents.generation.allowed_models` (default `[]` meaning "use the session's default model"), `external_agents.generation.implicit_trigger_enabled` (default `true`).
- **Security**: the `adapter-generator` agent type is *the* most security-sensitive new code path ŌĆö its output gets stored on disk and may run arbitrary binaries. Mitigations: (a) generator's tool surface is allowlisted in code, not in YAML; (b) `Bash` invocations from the generator are filtered by a generator-scoped command inspector that allows only `which`/`<bin> --version`/`<bin> --help`/`<bin> -h`/`<bin> help`/`man <bin>`/`cat <safe-path>` patterns; (c) drafts never enter the live registry until approval; (d) sandbox smoke test runs from `<profileDir>/external-agents/.drafts/` not from the live dir; (e) the generated YAML is rejected if its `binary` field resolves to a path the user did not explicitly name in the originating `agent_setup` call; (f) dangerous-command pattern scan on every generated `extra_args` and `env_override` string value before draft is offered for approval; (g) explicit refusal to generate adapters for system shells (`bash`, `sh`, `zsh`, `dash`, `csh`, `fish`).
- **Dependency on this change**: depends on `add-external-agent-tool` having merged. The `adapter-generator` agent type cannot be authored before the schema from Change 1 is fixed; the draft ŌåÆ publish move depends on the registry's hot-reload behavior from Change 1; the smoke runner is reused verbatim from Change 1.
- **Out of scope**: cross-machine sharing of generated adapters (one machine's `gemini` may be a different binary than another's); auto-regeneration on CLI version change (drift is detected and logged, not acted on); generation for `cli_tool`-class adapters (PATH scan already covers them ŌĆö no adapter needed).
