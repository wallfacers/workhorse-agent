## ADDED Requirements

### Requirement: FTS5 availability assertion at boot

The system MUST verify on startup that the linked SQLite build has FTS5 compiled in, by issuing `SELECT sqlite_compileoption_used('ENABLE_FTS5')` against the live connection. If the result is 0 the server MUST refuse to start, with an error message that identifies the failing capability and points the user at the issue tracker.

#### Scenario: FTS5 present, server starts normally

- **WHEN** the SQLite build returns 1 for `sqlite_compileoption_used('ENABLE_FTS5')`
- **THEN** the server proceeds with normal startup and migrations run as usual

#### Scenario: FTS5 absent, server refuses to start

- **WHEN** the SQLite build returns 0 for the same query
- **THEN** the server exits non-zero before opening the HTTP listener and writes a clearly-identified error mentioning FTS5 to stderr

### Requirement: FTS index over messages content

The system SHALL maintain an FTS5 virtual table `messages_fts` whose rows mirror `messages` rows. Content indexed for each message MUST be the concatenated text portion only: each `ContentBlock` of type `text` contributes its text; blocks of type `tool_use`, `tool_result`, `image`, or any other non-text block contribute nothing. The mapping from `messages.content_json` to the indexed string MUST be performed by a custom SQLite function registered by the Go driver setup.

#### Scenario: Inserted message indexed via trigger

- **WHEN** a new row is inserted into `messages` with a text content block
- **THEN** a corresponding `messages_fts` row is written within the same transaction, containing only the text portion of `content_json`

#### Scenario: Tool-use blocks not indexed

- **WHEN** a `messages` row contains only `tool_use` or `tool_result` blocks
- **THEN** the corresponding `messages_fts` row's content is the empty string

#### Scenario: Deleted message removed from index

- **WHEN** a row is deleted from `messages`
- **THEN** the corresponding `messages_fts` row is removed within the same transaction

#### Scenario: Updated message re-indexed

- **WHEN** an existing `messages` row's `content_json` is updated
- **THEN** the corresponding `messages_fts` row is rewritten within the same transaction

### Requirement: One-time backfill on migration

The migration that introduces `messages_fts` MUST populate it from existing `messages` rows in the same migration transaction. The backfill MUST complete before the server begins accepting any requests. Progress MUST be logged at least every 10,000 rows to stderr.

#### Scenario: Fresh install backfill is a no-op

- **WHEN** the migration runs against a database with zero existing `messages` rows
- **THEN** the migration completes without writing any FTS rows and without emitting progress logs

#### Scenario: Backfill of pre-existing messages

- **WHEN** the migration runs against a database containing N existing `messages` rows
- **THEN** after the migration completes, `messages_fts` contains exactly N rows whose content matches the text portion of the corresponding `messages.content_json`

### Requirement: session_search tool surface

The system SHALL expose a `session_search` tool that runs full-text search against `messages_fts` and returns raw matched messages plus surrounding context. The tool MUST NOT perform any LLM-based summarization, ranking rewrite, or content rewriting of returned matches.

#### Scenario: ASCII query passes through to FTS5

- **WHEN** `session_search` is called with an ASCII-only `query` field
- **THEN** the query is passed to `messages_fts MATCH ?` unchanged, and results are ordered by FTS5 rank

#### Scenario: Result shape includes context messages

- **WHEN** `session_search` returns one or more hits with `context_before: 3, context_after: 3`
- **THEN** each hit object contains `{session_id, message_idx, role, snippet, context_before: [...3 msgs], context_after: [...3 msgs]}` where `snippet` is generated by SQLite's `snippet()` function bounded to ≤ 30 characters and `context_*` arrays contain raw adjacent messages from the same session

#### Scenario: Limit and context defaults

- **WHEN** `session_search` is called without `limit`, `context_before`, or `context_after`
- **THEN** the defaults applied are `limit: 10`, `context_before: 5`, `context_after: 5`; `limit` is capped at 50 and context bounds are capped at 20

#### Scenario: No LLM call made

- **WHEN** `session_search` executes any query, regardless of result count
- **THEN** the tool performs no LLM API calls and emits no provider-related telemetry

### Requirement: CJK query handling

The system MUST handle queries containing CJK characters via a query-layer adaptation, not a custom tokenizer. For each run of consecutive CJK characters in the query: if the run is ≥ 3 characters long, the system rewrites it into trigram-style FTS5 phrase clauses; if the run is shorter than 3 characters, the system falls back to a `LIKE '%fragment%'` scan over the source `messages` table for that fragment instead of using FTS5.

#### Scenario: Long CJK query uses FTS5 trigram synthesis

- **WHEN** `session_search` is called with `query: "数据库迁移"` (5 CJK characters in one run)
- **THEN** the tool synthesizes a trigram FTS5 query (e.g. matching "数据库" AND "据库迁" AND "库迁移") and executes it via `messages_fts MATCH`

#### Scenario: Short CJK query falls back to LIKE

- **WHEN** `session_search` is called with `query: "迁移"` (2 CJK characters)
- **THEN** the tool runs a `LIKE` scan over the relevant `messages` rows and ranks results by recency

#### Scenario: Mixed ASCII and CJK query

- **WHEN** `session_search` is called with `query: "sqlite 迁移失败"`
- **THEN** the tool combines an ASCII MATCH clause for `sqlite` with the CJK strategy for `迁移失败`, joined by FTS5 `AND` if both are FTS-eligible, else degraded to LIKE across the whole query

### Requirement: Search scope and session-tree visibility

The default search scope for `session_search` SHALL be the conversation tree of the current session: the current session, all its ancestors (transitive via `parent_id`), and all its descendants. An explicit `scope` parameter MAY widen this to `"all"` (every non-deleted session in the database) or narrow it to `"session"` (current session only). Soft-deleted sessions (non-null `deleted_at`) MUST NEVER appear in results regardless of scope.

#### Scenario: Default scope covers session tree

- **WHEN** `session_search` is called from inside a session that has a parent and one child, without specifying `scope`
- **THEN** the search includes messages from the current session, its parent, and its child, but excludes any sibling session that shares only the parent

#### Scenario: Explicit all scope crosses trees

- **WHEN** `session_search` is called with `scope: "all"`
- **THEN** the search includes messages from every non-deleted session in the database

#### Scenario: Soft-deleted sessions excluded

- **WHEN** any session matching the scope filter has a non-null `deleted_at`
- **THEN** its messages MUST NOT appear in the result set, even with `scope: "all"`

### Requirement: Search ordering and pagination

Results returned by `session_search` MUST be ordered primarily by FTS5 rank (best match first) and secondarily by `messages.created_at` descending. When `limit` is reached the tool MUST return a `truncated: true` flag indicating that more matches existed but were not returned.

#### Scenario: Truncation flag set when more results exist

- **WHEN** a search would produce 12 hits but `limit: 10` is in effect
- **THEN** the response contains exactly 10 hits and `truncated: true`

#### Scenario: Truncation flag clear when all matches returned

- **WHEN** a search produces 4 hits with `limit: 10`
- **THEN** the response contains 4 hits and `truncated: false`
